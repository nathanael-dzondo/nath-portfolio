1. Les conceptions non sécurisées
Ensemble des méthodes qui vont permettent d'analyser régulièrement le code source d'une application afin de prévenir certaines attaques. Par déduction, une conception non sécurisée expose le code d'une application à certaines violations par un pirate.

Le meilleur moyens pour sécuriser est d'intégrer la sécurité dans un cycle de vie du développement d'une application (il faut penser à la sécurité lors de la création de l'application et lors des mis à jour de celle-ci).

2. Les injections
Une application est sensible aux injections lorsque les données fournies par l'utilisateur ne sont pas validées ou nettoyées avant leur traitement.

Bien que la plus populaire soit l'injection SQL, ces failles se produisent lorsque des données non fiables sont envoyées à un interprète. L'objectif du pirate est de faire en sorte que l'application est un comportement qui ne corresponde pas à son rôle.

Comme prévention, il faut toujours vérifier les données entrer par un utilisateur et protéger les caractères spéciaux.

3. Contrôle d’accès défectueux
Mécanisme qui permet de vérifier qu'un utilisateur accède uniquement à certaines ressourcess. Il empêche un utilisateur d'avoir accès à celui d'autrui car les conséquences peuvent être graves (divulgation d'informations privées, vol d'identité...).

Pour éviter cela, il serait préférable de mettre en place quelques,éléments pour durcir le contrôle d'accès (identification, authentification et autorisation). Aussi, sauvegarder dans les fichiers journaux les évènements de contrôle d'accès qui peuvent aider en cas de cyberattaque ; appliquer le principe du moindre privilège : les utilisateurs ou programmes ont des accès basique; et si possible utiliser le scanner OWASP ZAP qui permet de détecter les failles.

4. Le manque d'intégrité des données et des logiciels
Cette défaillance peut modifier ou supprimer des données de manière non autorisée et ainsi perturbées la sérialisation, qui est procédé qui consiste à prendre les objets dans le code de l'application et à les convertir dans un format qui est utilisé par la suite ; et la désérialisation qui consite à reconvertir les données séréalisées en objets utilisables par une application (exemple sauvegarde dans les jeux vidéos)

Pour éviter cela, toujours vérifier l'intégité des fichiers avant d'effectuer une mise à jour, utiliser un cryptage pour protéger les données sensibles en transcrit et au repos et enfin éviter la sérialisation des données provenant des sources non fiables.

5. Mauvaise configuration de la sécurité
Elles sont nombreuses : messages d'erreurs, utilisation des mots de passes sur les équipements; l'utilisation des fichiers xml sans vérifier le contenu;le traitement des fichiers xml sans vérification de contenu sur une application via le serveur peut permettre au pirate d'injecter du code qui va faire que le fichier déposer sur le serveur va retourner certaines informations qui vont le permettre de détourner le fonctionnement de l'application. On a aussi l'activation des fonctionnalités non utilisées comme par exemple laisser des port 80 et 21 activés.

Comme prévention, désactiver les services non utilisés avant les procédures de durcissement des systèmes d'exploitations

6.Les échecs cryptographiques et exposition de données sensibles
Elles concerne deux types de données : données stockés ( qui sont dans la machine) et les donnée sensibles (données stockées qu'on met à jour sur le serveur web). Ces données doivent être bien sécurisées lorsqu'elles sont transmises d'une machine à une autre (d'où la sensibilité des API). la collecte de carte bancaire par une application en HTTP est très vulnérables car le HTTP n'utilisent pas de chiffrement et peut donc être intercepté par un hacker. Trois principes sont donc recommandés à cela :

La classsification des données afin de connaître les données sensibles à chiffrer ou protéger.

Ensuite on a le chiffrement des données lors du transport ou le chiffrement des données transmises qu'on peut faire par le protocole HTTPS.

Enfin, éviter de stocker les données sensibles à moins que cela ne soit inévitable. Il existe aussi des scanners qui permettent de vérifier la version tls utilisée par un serveur pour vérifier s'il y'a des vulnérabilités tels que SSLLAB et SSLYze (sur le serveur local)

7.Composants obsolètes
Ils peuvent surgir lors de la conception d'une application par le développeur. Ces éléments constituent une porte d'entrer aux hackers.

Pour se prémunir, il faut avoir une procédure de mis à jour des composants ; désactiver les composant inutilisables ; suivre les publications sur les vulnérabilités et enfin télécharger les composants à partir d'une source fiable.

8.La supervision et la journalisation insuffisante
Le manque de supervision et de journalisation d'une application rend difficile la détection d'incident, d'intruison et la correction d'une anomalie.

Pour éviter cela, l'application doit envoyer ses fichiers vers un serveurs de log.

9. Falsification des requêtes côté serveur
Elle permet au pirate de forcer l'application à effectuer des requêtes vers un domaine ou serveur spécifique.

Comme prévention, il faut journaliser le trafic, implémenter un ensemble d'action, vérifier les informations envoyer par l'utilisateur et utiliser une liste blanche qui contiendra les protocoles et adresses IP autorisés

10. Identification et authentification de mauvaise qualité
La mauvaise gestion des identifications et des authentifications des utilisateurs constitue un danger pour un système d'information comme par exemple une application qui ne bloque pas pendant un certains temps les utilisateurs qui échouent plusieurs fois à l'authentification est une porte ouverte aux pirates pour utiliser "l'attaque par force brute".

Le mieux à faire est de changer le mot de passe par défaut ; journaliser les tentatives d'authentification et mettre en place un temporisateur après la succession d'authentification échouer.

Vulnérabilités des API
Les interfaces de programmation d'application (API) jouent un rôle crucial dans les applications modernes, mais elles présentent également des risques spécifiques en matière de sécurité.
